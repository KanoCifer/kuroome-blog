# 后端代码架构检查与重构报告
> 生成时间: 2026-03-26
> 最后更新: 2026-03-26
> 项目: ReadingList FastAPI 后端

## 总体评估
当前FastAPI后端功能完整，覆盖用户认证、书籍管理、微信读书导入、博客、RSS、AI助手等全功能模块，但架构存在明显的分层缺失和耦合问题：
- **核心痛点**：路由层承担过多职责（业务逻辑、数据访问、事务控制、IO处理），缺乏服务层/领域层/仓储层抽象
- **技术债务**：存在多处高危运行时错误、异步/同步边界混用、事务边界混乱
- **可维护性**：代码重复率高，单元测试困难，新功能开发成本高
- **稳定性**：多处阻塞调用和错误异常处理可能导致事件循环卡住或数据不一致

| 文件路径                            | 修复内容                                                                |
| ----------------------------------- | ----------------------------------------------------------------------- |
| `dependencies/csrf.py:61`           | `except SignatureExpired, Exception:` → `except (SignatureExpired, Exception):` |

| `routers/auth.py:677,966,1007`      | 多处旧式except语法修复为Python3元组写法                                |

### 2. DNS解析阻塞事件循环 ✅
- `rss.py`中`socket.getaddrinfo`已正确放入`asyncio.to_thread`线程池执行
- 修复了异步函数调用缺少`await`的类型错误
- 优化了导入结构

### 3. 显式事务提交冲突 ✅
- 移除所有路由中的显式`session.commit()`/`session.rollback()`共21处
- 事务完全由`get_session`依赖统一管理

### 4. 用户业务层拆分 ✅
新增文件：
| 文件                              | 说明                                               |
| --------------------------------- | -------------------------------------------------- |
| `repositories/__init__.py`        | 仓储层包导出                                       |
| `repositories/user_repository.py` | 数据访问层：User/Profile/PasskeyCredential所有查询 |
| `services/__init__.py`            | 服务层包导出                                       |
| `services/user_service.py`        | 业务逻辑层：认证、注册、Passkey、OAuth、资料管理   |

重构文件：
| 文件             | 变化                                         |
| ---------------- | -------------------------------------------- |
| `routers/auth.py`  | 1116行 → ~500行，移除所有直接DB查询和业务逻辑 |
| `routers/users.py` | 249行 → ~70行，移除所有直接DB操作和文件处理   |

架构分层（用户域）：
```
routers/auth.py, users.py  ← HTTP层：参数校验、Cookie/CSRF、响应构建
        ↓ Depends(DI)
services/user_service.py    ← 业务逻辑层：认证、注册、Passkey、OAuth
        ↓
repositories/user_repository.py ← 数据访问层：SQLAlchemy查询封装
        ↓
models/models.py            ← 领域模型：User, Profile, PasskeyCredential
```

## 二、待完成的重构任务

### 1. 其他业务域服务层拆分 ⬜
- [ ] `BookService`/`BookRepository`：封装书籍管理、微信读书导入逻辑
  - 涉及文件：`routers/books.py`（406行）、`routers/weread.py`
  - 当前状态：路由直接包含SQLAlchemy查询/写入/分页逻辑
- [ ] `BlogService`/`PostRepository`/`CategoryRepository`：封装博客、跨DB聚合逻辑
  - 涉及文件：`routers/blog.py`、`routers/admin.py`
  - 当前状态：路由中混合SQL（Category）与Mongo（Post）查询
- [ ] `RssService`/`RssRepository`：封装RSS订阅、抓取、解析逻辑
  - 涉及文件：`routers/rss.py`（647行）
  - 当前状态：路由承担HTTP拉取、解析、SQL写入、缓存写入
- [ ] `TodoService`：封装待办事项、Redis缓存/锁逻辑
  - 涉及文件：`routers/todos.py`
  - 当前状态：Redis操作直接在路由中执行

### 2. 跨DB聚合耦合 ⬜
**问题**：博客、RSS等业务同时操作PostgreSQL和MongoDB，聚合逻辑散落在路由中
**涉及文件**：`routers/blog.py`、`routers/rss.py`
**方案**：在服务层实现跨DB聚合逻辑，明确一致性策略

### 3. 路由间直接导入 ⬜
**问题**：`routers/monitor.py`直接导入`routers/admin.py:get_admin_user`
**方案**：将`get_admin_user`提取到`AdminService`

### 4. Pydantic Schema未被充分利用 ⬜
**问题**：大量路由手动构造响应字典，与已定义的Schema重复
**方案**：统一使用`schema.model_validate()`进行模型到DTO的转换

### 5. weread.py异步/同步边界错误 ⬜（高危）
**问题**：`import_books_from_weread`是async函数但被`asyncio.to_thread`调用
**方案**：改为直接`await`或注册为Taskiq后台任务

### 6. 中优先级优化点 ⬜
- 异步/同步边界统一（图片压缩、Feed解析）
- Redis序列化统一
- 异常处理统一（业务异常基类）
- 依赖注入优化（合并redis/redis2）
- 后台任务解耦

### 7. 低优先级改进 ⬜
- 将`User.get_real_ip`移到工具层
- 减少全局`settings`/`logger`直接导入
- 实现缓存失效统一策略
- 引入单元测试框架
