# 服务 / 仓储层解耦操作指南

> 基于 `user` 与 `rss` 两次重构整理。
> 目标：把路由层里的业务逻辑、数据访问和跨资源流程拆出去，形成稳定的 `router -> service -> repository` 分层。

## 1. 什么时候该拆

满足下面任意一条，就该考虑拆分：

- 路由函数里出现直接数据库操作（SQLAlchemy / Beanie / Redis）
- 路由函数里混有多个业务步骤：校验、查询、写入、缓存、任务调度
- 同一段逻辑在多个路由里重复
- 路由函数超过 80 行，且大部分不是 HTTP 参数处理
- 一个接口同时碰 SQL + Mongo + Redis + task

## 2. 三层职责边界

| 层级 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| Router | 参数接收、依赖注入、HTTP 状态码、响应包装、认证/鉴权 | 业务规则、SQL/Beanie 细节、跨表/跨库流程 |
| Service | 业务编排、事务边界、跨 Repository 协调、领域异常、外部 IO 编排 | FastAPI Response、路由参数格式、具体 SQL 语句 |
| Repository | 纯数据访问：查询、写入、删除、分页、条件封装 | `current_user`、HTTP、task、缓存、业务判断 |

### 核心原则

- **Router 只关心请求和响应**
- **Service 只关心业务结果**
- **Repository 只关心数据怎么拿/怎么存**

## 3. 标准迁移顺序

### Step 1：冻结接口行为

先不要改返回结构。

- 保持 URL、参数、响应 schema 不变
- 保持错误码语义不变
- 保持缓存 key、TTL、任务触发方式不变（除非你明确要改）

### Step 2：先抽 Repository

把所有“只跟表/集合有关”的代码先搬到仓储层。

典型动作：

- `session.execute(...)`
- `select(...) / update(...) / delete(...)`
- Beanie 的 `find / save / delete`

### Step 3：再抽 Service

把路由里的业务流程搬到服务层：

- 组合多个 Repository
- 做重复检查、权限检查、状态转换
- 统一抛业务异常
- 编排外部 IO（抓取、解析、同步、刷新）
- Redis 的基础读写，放在 Service 里


### Step 4：让 Router 只剩 HTTP glue

Router 最终应该只保留：

- `Depends(...)`
- 参数校验/转换
- 调用 service
- 把 service 异常映射成 `APIResponse.error(...)`
- 返回 `APIResponse.ok(...)`

### Step 5：补测试【可选】暂时不需要

- repo：测查询条件和写入结果
- service：测业务规则和异常
- router：测入参、状态码、响应结构

## 4. User 域：最适合的拆分模板

user 域已经是一个很好的参考：

- `backend/app/api/v1/auth.py`
- `backend/app/api/v1/users.py`
- `backend/app/services/user_service.py`
- `backend/app/repositories/user_repository.py`

### 这个模板的特点

- Router 负责 `current_user`、Cookie、CSRF、响应格式
- Service 负责认证、注册、Passkey、OAuth、资料更新
- Repository 负责 `User / Profile / PasskeyCredential` 的 SQLAlchemy 操作

### 推荐签名风格

```python
class UserRepository:
    def __init__(self, session: AsyncSession) -> None: ...

    async def get_user_by_email(self, email: str) -> User | None: ...
    async def create_user(self, data: UserCreate) -> User: ...


class UserService:
    def __init__(self, repo: UserRepository) -> None: ...

    async def register_user(self, data: UserCreate) -> User: ...
    async def update_profile(self, user_id: int, data: ProfileUpdate) -> User: ...
```

### User 域的迁移要点

- `current_user` 只出现在 Router
- Repository 只收 `user_id: int`、`email: str` 这类原始值
- Service 抛 `ValueError / RuntimeError` 可以工作，但更推荐业务异常类

## 5. RSS 域：从“胖路由”到服务编排

RSS 是更复杂的模板，因为它混了：

- SQLAlchemy
- MongoDB / Beanie
- Redis 缓存
- Feed 抓取与解析
- Taskiq 任务

### 建议拆分方式

**Router（`backend/app/api/v1/rss.py`）**

- 接收 URL、分页、read/unread、refresh、delete 请求
- 处理 `APIResponse`
- 处理缓存命中/失效
- 把业务异常转成 HTTP 响应

**Service（`backend/app/services/rss_service.py`）**

- 抓取并解析 feed
- 校验订阅重复
- 刷新订阅
- 读取/更新文章状态
- 组合 SQL + Mongo 的业务流程

**Repository（`backend/app/repositories/rss_repo.py`）**

- `get_subscription_by_id()`
- `get_user_subscriptions()`
- `is_user_subscribed_to_feed()`
- `delete_subscription()`

### RSS 的关键经验

1. **Repository 不认识 `current_user`**
   - 错误：`RssInfo.user_id == current_user.id`
   - 正确：`user_id: int` 作为参数传入

2. **Service 负责业务判断**
   - 例如：重复订阅、权限检查、文章可读性、刷新流程

3. **Router 只做 HTTP 映射**
   - 例如：`RssDomainError` → `APIResponse.error(...)`

4. **外部 IO 要有边界**
   - 抓取、解析、保存文章可以放进 Service
   - 但不要把这些逻辑散在多个路由里

### RSS 的典型调用链

```text
POST /rss/parse-rss
  -> Router 解析参数 + 查缓存
  -> Service 抓取 feed + 解析 + 保存订阅/文章
  -> Router 写缓存 + 返回 APIResponse
```

```text
GET /rss/articles
  -> Router 获取分页参数
  -> Service 组织文章查询
  -> Repository 做 SQL 查询
  -> Router 返回结果
```

## 6. 函数签名怎么切

### Repository

Repository 只收“简单值”或“纯 DTO”：

- `int`
- `str`
- `bool`
- `list[str]`
- 领域输入对象（如果它不依赖 FastAPI）

不要传：

- `Request`
- `Response`
- `current_user`
- `Depends(...)` 产物

### Service

Service 方法尽量表达业务动作：

- `create_xxx`
- `refresh_xxx`
- `delete_xxx`
- `mark_read`
- `list_articles`

Service 可以：

- 调多个 repository
- 调外部 client
- 抛业务异常

### Router

Router 做三件事就够了：

1. 拿参数
2. 调 service
3. 包装响应

## 7. 常见反模式

- 在 Repository 里 import `current_user`
- 在 Router 里直接写 SQLAlchemy / Beanie 查询
- 在 Router 里 `task.kiq().wait_result()` 阻塞等待
- Service 直接返回 FastAPI `JSONResponse`
- 一个函数同时做参数校验、查询、写库、发任务、写缓存
- 把跨库聚合逻辑散在路由里
- 用 `dict` 到处手写响应结构，绕开 schema

## 8. 推荐的落地清单

迁移一个模块时，按这个顺序做：

- [ ] 找出所有直接 DB / Redis / Taskiq 调用点
- [ ] 列出重复逻辑和跨资源逻辑
- [ ] 先抽 Repository
- [ ] 再抽 Service
- [ ] Router 只保留 HTTP 与响应包装
- [ ] 统一业务异常类型
- [ ] 保持响应 schema 不变
- [ ] 补单元测试 / 集成测试
- [ ] 跑 `ruff format`、`ruff check`
- [ ] 跑目标测试并检查 `lsp_diagnostics`

## 9. 什么时候可以不拆

如果一个接口只是：

- 单表单行查询
- 没有业务规则
- 没有缓存/任务/跨库

可以先不引入复杂抽象。

但只要它开始变胖，优先按上面的模板拆成 `service + repository`。

## 10. 最小模板

```python
class XxxRepository:
    def __init__(self, session: AsyncSession) -> None:
        self.session = session

    async def get_by_id(self, id_: int) -> Xxx | None:
        ...


class XxxService:
    def __init__(self, repo: XxxRepository) -> None:
        self.repo = repo

    async def do_action(self, user_id: int, payload: XxxIn) -> XxxOut:
        ...


@router.post("/xxx")
async def create_xxx(
    payload: XxxIn,
    current_user: User = Depends(get_current_user),
    service: XxxService = Depends(get_xxx_service),
) -> APIResponse:
    result = await service.do_action(current_user.id, payload)
    return APIResponse.ok(data=result)
```

## 11. 一句话总结

**Router 管 HTTP，Service 管业务，Repository 管数据。**

user 域是这个模式的标准模板，rss 域是这个模式的复杂版模板。
